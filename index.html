    • Eddie's R Code for Fitting the Model Migration Schedule with Student Peak

      (migration, R code)

      Model migration age schedules have been developed and described over the past few decades (a great resource: Rogers and Castro, 1981), and they are useful for understanding human migration, which has some recurrent patterns for childhood, labor force ages, retirement ages, and old ages. Tom Wilson of Charles Darwin University extended the Rogers-Castro migration model, adding a very useful student curve, and also provides terrific guidance and instruction on the overall model. This R code uses the guidance and instruction from Wilson's model migration schedules with student peaks paper and the related SPMMS Excel workbook shared on this page, along with R's random number generator, and sampling (Monte Carlo) from recursively smaller uniform distributions of model parameters to locate a best fit for each of the components. The sampling-based fitting process is just my own "DIY" solution (I haven't been able to find another example or reference for fitting-by-sampling done in this way, or an evaluation of it - please let me know if you know one, or if you have insights that might be instructive or generally helpful), and it hasn't been carefully tested at all, but it's neat it appears so far to do the job pretty well (if slowly), and I think relying on uniform distribution bounds rather than single values for starting-parameter selection may make the overall process of fitting the model a bit easier. To make the code work, you should be able to simply select-all, copy, and paste into an R command line (the example input data is linked through the Internet, and can be simply downloaded for review). Copies of the code with some application of and comparison to R's nls() and lm() functions, and to more plain Monte Carlo, are available here. –August 2018 (updated December 2018) ^

    • Chuck Coleman's Constraining Arrays Website and SAS Code

      (population estimates, surveys, SAS code)

      From the related paper: "Many applications require constraining arrays of numbers to controls in one or two dimensions. Example applications include survey estimates, disclosure avoidance, input-output tables, and population and other estimates and projections. If the results are allowed to take on any nonnegative values, raking (a.k.a. scaling) solves the problem in one dimension and two-way iterative raking solves it in two dimensions. Each of these raking macros has an option for the user to output a dataset containing the rakes. The problem is more complicated in one dimension if the data can be of any sign, the so-called "plus-minus" problem, as simple raking may produce unacceptable results. This problem is addressed by generalized raking, which preserves the structure of the data at the cost of a nonunique solution. Often, results are required to be rounded so as to preserve the original totals. The Cox-Ernst algorithm accomplishes an optimal controlled rounding in two dimensions. In one dimension, the Greatest Mantissa algorithm is a simplified version of the Cox-Ernst algorithm." Chuck also notes that the gras.sas macro implements the Generalized RAS algorithm, which rakes matrices of mixed sign. A great resource. –July 2016 ^

    • Hauer, Baker, and Brown's Implied Total Fertility Rate Calculator

      (fertility, Excel spreadsheet)

      From the related abstract: "Indirect estimation methodologies of the total fertility rate (TFR) have a long history within demography and have provided important techniques applied demographers can use when data is sparse or lacking. However new methodologies for approximating the total fertility rate have not been proposed in nearly 30 years. This study presents a novel method for indirectly approximating the total fertility rate using an algebraic rearrangement of the general fertility rate (GFR) through the known relationship between GFR and TFR. It then compares the proposed method to the well-known Bogue-Palmore method. These methods are compared in 196 countries and include overall errors as well as characteristics of the countries that contribute to fertility behavior. Additionally, these methods were compared geographically to find any geographical patterns. We find this novel method is not only simpler than the Bogue-Palmore method, requiring fewer data inputs, but also has reduced algebraic and absolute errors when compared with the Bogue-Palmore method and specifically outperforms the Bogue-Palmore method in developing countries. We find that our novel method may be useful estimation procedure for demographers." Matt Hauer (University of Georgia), Jack Baker (University of New Mexico), and Warren Brown (Cornell) put the project together and it's very useful. –March 2014 (December 2018) ^

    • Eddie's Employment-Based Population and Housing Projection Code

      (population projections, R code)

      Planners are often required to prepare or acquire long-term population, employment and housing projections for their areas, and population projections with integrated employment or housing can be a heavy lift. I had the thought that it shouldn't be too tricky though to just modify R code for Basic Population Projection and make net-migration employment-based with some broadly used parameters (that economists have strong familiarity with or ability to access quickly), and then simply calculate housing ex-post, and in fact it wasn't too tricky. Total migration counts are scaled based on working-age migration counts. To see the code work, and a selection of output-graphics, you should be able to simply select-all of the code, copy, and paste into an R command line (input data (csv format) is linked through the Internet, and can be simply downloaded for review). I don't know of other simple and readily available code to do this (please let me know if you have some). The economic and housing inputs are broad, so anyone (regional economists, local experts, etc.) could provide them. The example input data were not carefully considered and are not meant to be useful for any certain area. Here are the employment and housing variables: TOTAL CIVILIAN EMPLOYMENT, LABOR FORCE PARTIPATION RATE, UNEMPLOYMENT RATE, COMMUTER RATE, WOMENS SHARE OF TOTAL CIVILIAN EMPLOYMENT, PERSONS PER HOUSEHOLD, and VACANCY RATE. As I always explain, there isn't general evidence that using employment as an input will improve predictive power for population projections (see meta-analysis by Stan Smith of the University of Florida on this), but it can be useful to have more output variables to assess. Here is an implementation of it (with Shiny) (and an additional copy). If you have any questions, don't hesitate to shoot me an email at edyhsgr@gmail.com. –March 2014 (updated December 2018) ^

    • Eddie's Brass Relational Logit Mortality Model Code

      (mortality, R code)

      I think William Brass' most famous contribution to demography (he had many, and is surely among the most productive and revered demographers ever) is the Brass Relational Logit Model for mortality. The Brass Relational Logit Model is used to simply and reasonably adjust the level and shape of a life table's lx (survivorship) curve, based on the level and shape of another life table's lx curve. This ability for a reasonable adjustment by two parameters is useful for many things, including (1) life table estimation with limited data, where a life table with very large margins of error and an unreliable pattern across age can be modeled with a life table that has very small margins of error, and (2) life table forecasting, in which the the level and shape terms (alpha and beta, respectively) can be projected over time. Brass alpha in particular has been a subject of interest for mortality forecasting, as it is believed to change linearly over time for many cases/periods (note rectangularlization though) (Brass beta is believed to often stay trendless-- I think particularly in developed regions). Linked above is an R function/script to apply the Brass Relational Logit model, and make selected output graphics. To make it work, you should be able to simply select-all of the code, copy, and paste into an R command line (example input data (csv format) is linked through the Internet, and can be simply downloaded for review).
